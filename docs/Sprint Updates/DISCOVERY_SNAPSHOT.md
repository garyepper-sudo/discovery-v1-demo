# Current Identity

Discovery is an **Organizational Decision Intelligence Platform** built upon a canonical Organizational Understanding Engine.

Its purpose is to transform fragmented organizational evidence into durable understanding, explain how organizational systems influence one another, evaluate intentional executive interventions, and help leaders reason about future organizational outcomes.

Discovery is not a dashboard.

Discovery is not a chatbot.

Discovery is an **Adaptive Organizational Reasoning System**.

Discovery now:

* understands an organization's current state,
* explains why that state exists,
* predicts its most likely future,
* evaluates whether prior predictions became reality,
* converts prediction outcomes into canonical learning events,
* continuously learns from prediction accuracy,
* calibrates confidence through observed outcomes,
* represents executive decisions and intentional organizational interventions,
* generates plausible intervention options from executive objectives,
* constructs a canonical Organizational Causal Model,
* maps interventions to affected organizational conditions,
* propagates direct changes across multi-stage causal relationships,
* aggregates direct and indirect organizational effects safely,
* and validates decision reasoning through a canonical Decision Intelligence Experiment.

The first generation of Discovery's Cognitive Operating System remains complete.

Discovery now possesses:

* a closed adaptive learning loop,
* a canonical causal reasoning substrate,
* an executable organizational influence engine,
* and the first working version of an end-to-end Decision Intelligence pipeline.

Future development should focus on increasing the intelligence, accuracy, realism, and benchmark coverage of these existing systems rather than expanding architecture prematurely.

* one capability owner,

# Sprint Rules

Before implementing any capability ask:

1. Which Operating System owns it?
2. Which capability owns it?
3. Which cognitive object does it produce?
4. Is there already a canonical producer?
5. Does Runtime need to persist it?
6. Which later reasoning stages consume it?
7. Does Executive Projection need to expose it?
8. Does Atlas or a Decision Intelligence Experiment validate it?
9. Does the Capability Registry describe it?
10. Can an existing capability be extended?
11. Is this production cognition or benchmark-only scaffolding?
12. Will the implementation improve a measured reasoning dimension?

Do not couple shared cognitive systems directly to capability-specific internals when a canonical object or adapter can preserve architectural separation.

Do not expose internal reasoning complexity to users unless it produces immediate executive utility.

Do not add more architecture while the current reasoning pipeline remains untested.
